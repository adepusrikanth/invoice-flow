1. Architecture Overview

InvoiceFlow uses a modern, scalable architecture built on PostgreSQL for data persistence, Redis for caching and job queues, and a RESTful API built with Node.js (Express). The system follows a multi-tenant architecture where each user’s data is logically isolated using organization-level scoping.

1.2 Entity Relationship Overview

The database is organized around these core entities: Organizations (multi-tenant), Users, Clients, Invoices, Invoice Items, Payments, Expenses, and Recurring Schedules. Below is the relationship structure:

Organization (1) ─── (*) User

Organization (1) ─── (*) Client

Organization (1) ─── (*) Invoice

Client        (1) ─── (*) Invoice

Invoice       (1) ─── (*) InvoiceItem

Invoice       (1) ─── (*) Payment

Invoice       (1) ─── (0..1) RecurringSchedule

Organization  (1) ─── (*) Expense

User          (1) ─── (*) AuditLog

2.1 Key Indexes

-- Performance indexes

CREATE INDEX idx_invoices_org_status ON invoices(org_id, status);

CREATE INDEX idx_invoices_client ON invoices(client_id);

CREATE INDEX idx_invoices_due_date ON invoices(due_date) WHERE status != 'paid';

CREATE INDEX idx_invoices_number ON invoices(org_id, invoice_number);

CREATE INDEX idx_payments_invoice ON payments(invoice_id);

CREATE INDEX idx_clients_org ON clients(org_id) WHERE deleted_at IS NULL;

CREATE INDEX idx_expenses_org_date ON expenses(org_id, date);

CREATE INDEX idx_audit_entity ON audit_logs(entity_type, entity_id);

CREATE INDEX idx_recurring_next ON recurring_schedules(next_date) WHERE is_active;



-- Full-text search

CREATE INDEX idx_invoices_search ON invoices

  USING GIN(to_tsvector('english', invoice_number || ' ' || notes));

CREATE INDEX idx_clients_search ON clients

  USING GIN(to_tsvector('english', name || ' ' || email || ' ' || company));





3. REST API Design

The API follows REST conventions with JSON request/response bodies. All endpoints require authentication via Bearer token. Responses use standard HTTP status codes and include pagination for list endpoints.

3.1 Base URL & Authentication

Base URL:  https://api.invoiceflow.com/v1



Headers:

  Authorization: Bearer <api_key>

  Content-Type: application/json

  X-Organization-Id: <org_uuid>  (for multi-org users)



Rate Limits:

  Free:     100 requests/hour

  Starter:  1,000 requests/hour

  Pro:      5,000 requests/hour

  Business: 20,000 requests/hour



3.2 Standard Response Format

// Success

{

  "success": true,

  "data": { ... },

  "meta": { "page": 1, "per_page": 20, "total": 150 }

}



// Error

{

  "success": false,

  "error": {

    "code": "VALIDATION_ERROR",

    "message": "Invalid email address",

    "details": [{ "field": "email", "message": "must be valid" }]

  }

}

4. Webhooks

InvoiceFlow sends webhook events for key actions. Clients can register webhook URLs in their dashboard. Events are delivered with retry logic (3 attempts with exponential backoff).
